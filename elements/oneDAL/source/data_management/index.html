Data management¶

Effective data management is among key constituents of the performance of a data analytics application. For oneDAL, effective data management requires effectively performing the following operations:

  • raw data acquisition

  • filtering and normalization

  • data conversion to a numeric representation

  • data streaming to an algorithm

Key concepts¶

oneDAL provides a set of customizable interfaces to operate on out-of-memory and in-memory data in different usage scenarios, which include batch processing, online processing, and complex scenarios.

Table¶

To represent a dataset in a numeric data format, oneDAL introduces a table concept - an immutable tabular view of a dataset where table rows represent observations and columns represent features.

oneDAL defines a set of table types, each represent data in various data formats. See Tables section for more details about requirements of table types and table concept.

Data source¶

Data source defines an interface to extract data from raw sources like files, databases, etc. and to represent them as a table object. It can automatically transform non-numeric categorical and continuous values of data into a numeric representation.

It can also provide an abilities to filter or normalize feature values being extracted or recover missing values:

  • It is not a rare situation when only a subset of features can be measured at a given moment. In this case, the non-measured features in the feature vector become blank, or missing. Special statistical techniques enable recovery (emulation) of missing values.

  • When a given measurement introduces an outlier or when an observed object can produce an outlier by nature, special statistical techniques enable detection of outliers and recovery of the abnormal data.

Data are not always fit the local memory, especially when processing with accelerators. Data source provides an interface to load data by batches and orchestrates its efficient extraction directly into local memory of the device. Therefore data source enables complex scenarios of data analytics, like online computations.

Table builder¶

Some table objects require complex procedures to create them. To make easy-of-use table creation from data of various data formats or other objects of different types, oneDAL introduces a special builder concept for table types. See table builders section for details.

Accessor¶

There are a lot of data formats used in datasets to store the data. oneDAL defines different types of tables, table builders and other objects to support different logical types of data and various data formats. However, all these objects have different logic to access their underlying data.

Accessor allows:

  • to make uniform access to the data from various sets of objects of different types.

  • to perform efficient access to the data without exposing implementation details of a data container.

  • to give an ability to focus on the data acquisition in the desired data format with a wanted set of operations over the data.

See Accessors section for details.

Relations between concepts¶

Typical workflow with data in oneDAL consists of next steps:

  1. Select an appropriate data source for extracting out-of-memory data, or select a table builder to manually create a table object.

  2. Create table object - a numerical representation of data, with use of selected data source, table builder or table objetc constructor directly.

  3. Run a selected algorithm with created table object.

  4. Get algorithm results and access their data directly or using selected accessor.

Next sequence diagram briefly shows dependencies between table, builder, and accessor concepts.

Builder is responcible to create a table object, which data can be accessed either directly using table interface or with help of accessor object. Since the table object is immutable, this is not possible to change the data inside it.

To change the data of table object, its data should be moved inside builder object. Once it done, data inside builder can be changed by the accessor.

See sections about Tables, table builders, and Accessors for details.
